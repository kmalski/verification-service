# Verification Service

A Spring Boot application that orchestrates payment verification workflows across multiple independent providers. It
starts verifications in the background, executes checks concurrently, aggregates their results, and produces a final
decision.

Implemented checks:

- `VELOCITY` - rejects when the simulated number of customer payments exceeds the configured limit.
- `SANCTIONS` - rejects payments from sanctioned countries in the fake sanctions provider.
- `FRAUD` - approves, marks for review or rejects based on amount thresholds in the fake fraud provider.

## Tech stack

- Java 25 with preview features enabled
- Spring Boot 4
- Gradle
- JUnit 5
- ArchUnit

## Running locally

Bash:

```bash
./gradlew bootRun
```

PowerShell:

```powershell
.\gradlew.bat bootRun
```

The application starts on the default Spring Boot port: `http://localhost:8080`.

## API

### Start verification

```http
POST /verifications
Content-Type: application/json
```

Example body:

```json
{
  "payment": {
    "paymentId": "payment-1",
    "customerId": "customer-1",
    "amount": "100.00",
    "currency": "PLN",
    "country": "PL"
  }
}
```

Example response:

```json
{
  "verificationId": "a UUID generated by the service",
  "status": "QUEUED"
}
```

### Get verification

```http
GET /verifications/{verificationId}
```

Example response:

```json
{
  "verificationId": "a UUID generated by the service",
  "status": "COMPLETED",
  "decision": "APPROVED",
  "checks": [
    {
      "type": "VELOCITY",
      "status": "PASSED",
      "reason": null
    }
  ]
}
```

## Request scripts

The `requests/` directory contains small smoke-test scripts that exercise the API end to end. They:

1. Send `POST /verifications` with example payment data.
2. Read the returned `verificationId`.
3. Poll `GET /verifications/{id}` every second.
4. Stop when the verification reaches `COMPLETED` or `FAILED`, or fail after 60 seconds.

PowerShell:

```powershell
.\requests\verification.ps1
```

Bash (requires `curl` and `jq`):

```bash
./requests/verification.sh
```

## Project structure

- `domain` - value objects, verification aggregate, statuses, check results and decision policy.
- `application` - use cases, ports, orchestration and check strategies.
- `infrastructure` - REST API, in-memory persistence, fake provider adapters and configuration binding.
- `requests` - executable API smoke-test scripts.
